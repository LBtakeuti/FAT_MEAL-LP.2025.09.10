## F14-3: コラム記事 view_count の二重インクリメント懸念解消

### 概要
コラム記事詳細の閲覧数（`articles.view_count`）が、SSRと内部APIの両方でインクリメントされる実装になっている。
現状はSSRしか稼働していないため二重カウントは起きていないが、将来クライアントからAPIを呼ぶ実装に変えた場合に重複する。

### 背景
F14-2（コラムUIリニューアル）実装時のレビュー指摘事項。
- `app/blog/[slug]/page.tsx`（Server Component）で view_count を +1
- `app/api/blog/[slug]/route.ts`（API Route）でも view_count を +1
両方が同時に走るシナリオが現状はないが、将来の改修時に事故が起きる可能性がある。

### 対象ファイル
- `app/blog/[slug]/page.tsx`
- `app/api/blog/[slug]/route.ts`

### 対応方針（暫定）
- view_count のインクリメント責任を片方に寄せる
- 推奨: API ルート側のインクリメントを削除（SSR集約）
- もしくは共通ヘルパー（lib/blog/incrementViewCount.ts 等）に切り出し、重複呼び出しをガード

### 優先度
- 低（現状実害なし）
- ただしクライアント側で API を呼ぶ改修を入れる前に必ず先に対応する

## F5: 2027年以降の祝日リスト追加

### 概要
`lib/business-days.ts` に登録されている祝日定数を、2027年以降にも拡張する。

### 背景
F4-2（配送カレンダーの配置基準を「配送作業日」に変更）の実装時、
土日祝をスキップするロジックを導入した。祝日判定は `lib/business-days.ts` 内の
定数リストに依存しているが、現状の登録は **2024〜2026 年まで**。

2027年到来後にこのファイルを更新しないと、本来「祝日」として扱うべき日が
営業日として扱われ、配送カレンダーの自動配置が業務実態と乖離するリスクがある。

### 対象ファイル
- `lib/business-days.ts`

### 対応方針（暫定）
- 内閣府のHPまたは祝日マスタAPIから2027年以降の祝日リストを取得
- 直接ハードコードでもよいし、年度ごとに自動生成する仕組みでもよい
- 既存リストの形式に合わせて追記

### 優先度
- 中（2026年内に対応推奨）
- 2026/12/31 を過ぎても未対応の場合は優先度「高」に引き上げ

### 起点
- F4-2 実装時の指摘（2026-05-25）
